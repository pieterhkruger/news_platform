TESTING THE DAILY INDABA API IN POSTMAN
=======================================

This is a copy-and-paste guide for manually testing the API routes implemented
through daily_indaba/api_views.py.

No Postman environments are needed.
No Basic Auth is needed.
For protected routes, use Postman's Authorization tab -> API Key.


WHY THIS PROJECT USES DRF TOKEN AUTH AND NOT BASIC AUTH
-------------------------------------------------------

Basic Auth was used in M06T07, but this project is different.

This project is configured to use Django REST Framework token authentication.
You can see that from the code:

1. In news_platform/settings.py, REST_FRAMEWORK sets:
   DEFAULT_AUTHENTICATION_CLASSES = [
       "rest_framework.authentication.TokenAuthentication",
   ]

   That means protected API routes expect DRF token auth.

2. There is no BasicAuthentication configured there.
   So Postman Basic Auth will not satisfy the authentication layer.

3. In daily_indaba/api_views.py:
   - TokenLoginView subclasses ObtainAuthToken
   - the view imports Token from rest_framework.authtoken.models
   - on login it runs Token.objects.get_or_create(user=user)
   - on logout it runs request.user.auth_token.delete()

   That is a full login-token-logout flow, not a Basic Auth flow.


WHY TOKEN AUTH IS BETTER HERE
-----------------------------

For this project, token auth is better than Basic Auth because:

1. You send username and password only once, to /api/token/
2. After that, you send only the token on protected requests
3. The token can be deleted on logout through /api/token/logout/
4. It matches the code that this project is actually using

With Basic Auth, you would keep sending username and password on every
request. That is not the pattern this project is built around.


THE WORD "Token" MUST COME BEFORE THE KEY
-----------------------------------------
The Authorization header must look exactly like this:

Authorization: Token your_actual_token_here

The word "Token" is not something custom written in this project. It is DRF's
default keyword for TokenAuthentication.

In DRF, TokenAuthentication uses the keyword:

Token

This project does not override that default.

So if Postman sends:

Authorization: 123abc456def

that is wrong.

If Postman sends:

Authorization: Token 123abc456def

that is correct.

This is why on protected routes you must use:

Authorization tab -> API Key
Key: Authorization
Value: Token PASTE_TOKEN_HERE
Add to: Header

POSTMAN RULES FOR THIS PROJECT
------------------------------

RULE 1 - /api/token/ (LOGIN REQUEST)

Use:
- Method: POST
- Authorization tab -> No Auth
- Body tab -> raw -> JSON

Paste this JSON:
{
  "username": "lerato.maseko",
  "password": "demo1234"
}

Do not use Basic Auth for /api/token/.


RULE 2 - ALL PROTECTED REQUESTS

Use:
- Authorization tab -> API Key
- Key: Authorization
- Value: Token PASTE_TOKEN_HERE
- Add to: Header

Do not use:
- Authorization tab -> Basic Auth
- Authorization tab -> Bearer Token


RULE 3 - WRITE REQUESTS

For POST, PUT, and PATCH:
- Body tab -> raw -> JSON
- Paste the body shown in this document

When you choose raw JSON, Postman normally sets Content-Type automatically.
